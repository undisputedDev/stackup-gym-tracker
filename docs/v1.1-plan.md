# v1.1 Plan — clarity, trust, and a smarter core

Goal: make the one key feature (auto-progression) understandable in the first minute,
protect user data, and deepen the progression logic — without adding feature bloat.
Explicitly out of scope: rest timer, social, programming/periodization, gamification.

## Phase 1 — Understand it (session-screen clarity)

The arrow system is the product; new users must get it without trial and error.

1. **Label the rep chips.** A small hint label above the chip row: "Reps per set".
   The boxes are currently unlabeled and mean nothing until tapped.
2. **Show the consequence live.** When reps are entered and the mark flips, the card's
   hint line shows what it means concretely: "16 reps → next time 62,5 kg ▲"
   (colored like the mark). The arrow explains itself by showing its effect.
3. **One-time explainer.** On first opening a session: a small dismissible overlay
   ("How progression works": reps in range → keep · above → ▲ more weight next time ·
   below → ▼ less; tap the badge to override). Shown once (flag in settings,
   migration v5); re-openable via Settings → "How progression works".

## Phase 2 — Trust (data safety & correction)

4. **Backup & restore.** Settings → Data: "Create backup" (share a dated copy of the
   database file) and "Restore backup" (file picker, integrity check via PRAGMA,
   close connection, swap file, re-run migrations, confirm-overwrite dialog).
   Local-only app = phone loss currently destroys all history; this is the insurance.
5. **Delete/discard sessions.** In-progress session: toolbar "Discard session"
   (confirm → delete + leave). Historical read-only session: toolbar "Delete session"
   (confirm → delete, jump to neighbour or leave). Today a botched session is
   permanent and pollutes "last done" + charts.

## Phase 3 — Smarter core (deepen the key feature)

6. **Deload after repeated downs.** 3 consecutive ▼ sessions on an exercise → instead
   of another −2,5 kg, suggest a real reset: last weight × 0.9, rounded to the
   exercise's increment. Hint shows "Deload". Engine change: `SuggestNext` receives
   the last few entries (not just one); full unit tests.
7. **PR recognition.** Finish summary gets "★ New best: Bench Press 45 kg" lines when
   a logged weight beats that exercise's all-time best (weight-based; reps count for
   rep-based). One DB query per logged exercise at finish time.
8. **"Last time" on the card.** Hint line includes the previous session's actual
   result: "Target 10–15 · last 15/12/9 @ 60". Answers "what did I do last week?"
   at the moment it matters.

## Phase 4 — Design polish

9. **Redraw the lateral-raise icon** (current one reads cryptic at 22 px).
10. **Stats X-axis thinning.** Label step derived from the selected range
    (3M → 2 weeks, 1Y → 2 months, All → auto) so a year of data doesn't collide.
11. **Light-theme contrast pass.** Hint gray fails readability in bright light;
    bump to a darker gray in light mode (dark mode unchanged).

## Phase 5 — German localization (last before the store listing)

12. **resx-based strings** (`AppStrings.resx` + `AppStrings.de.resx`), all UI text
    moved out of XAML/VMs; device language decides.
13. **Preset names localized at seed time** (fresh installs on a German device get
    "Bankdrücken"/"Latzug" etc.); existing data is user data and stays untouched.
14. Store-listing texts DE/EN (when the Play Console is set up).

## Order & verification

Phases land as separate commits in the order above; each phase: tests where logic
changed (engine, backup validation, migrations), build Android + Windows, deploy to
emulator with screenshot verification, then commit.
